## 要求定義

### 課題1：出退勤の管理を紙で行っており、打刻も管理も手間がかかる
対応方針|要求
----|----
勤怠管理を簡素化する|2ステップ程度で打刻が完了する方式<br>手入力不要の勤怠管理を実現する


### 課題2：プライバシーへの懸念や、個人情報漏洩リスクが導入の障壁になる
対応方針|要求
----|----
個人情報の収集を最小限に留め、<br>端末内に情報を残さないセキュアな設計の徹底|氏名や住所をアプリ内に留めず、ID管理する仕組み<br>外部から通信内容を覗けないようにする

### 課題3：工場の奥まった場所など、不安定な通信環境では打刻できない可能性がある
対応方針|要求
----|----
オフライン・不安定な通信環境下での動作保証とデータ整合性の維持|オフライン時に端末にデータを一時保存、復帰時に自動同期する仕組み

### 課題4：画面操作に不慣れな従業員も使用する
対応方針|要求
----|----
迷わせないユーザ体験<br>最小アクションでの打刻完了|かざして、もしくは読み取ってログインするだけで打刻が完了する方式

### 課題5：QRコードのコピーなどで、事業所以外の場所で不正に打刻される可能性がある
対応方針|要求
----|----
打刻場所の正当性を担保|打刻時間のGPS位置情報取得し、現場の座標と照合
多重的な認証ロジックの導入|直接URLを打ち込んだらエラー画面を表示させるようにする


### 課題6：打刻データが給与管理等のソフトと互換性がなければ、管理者の負担は減らない
対応方針|要求
----|----
データ流通のシームレス化による、管理部門の転記作業ゼロ化|外部ソフトにインポートするためのCSV出力機能を実装<br>データのフォーマットを可変式にし、多くの外部ソフトに対応可能にする

### 課題7：不要な機能や複雑な料金形態では導入に至らない
対応方針|要求
----|----
必要最低限の機能実装|打刻、シフト作成、給与計算ソフトとの連携の3機能に絞って実装する
簡潔な料金形態の実現|初期費用を一度だけ請求し、以降は任意のサポート料金を月額で請求する

### 課題8：打刻時の操作ミスなどで不正な打刻データが保存される可能性がある
対応方針|要求
----|----
入力の段階でミスしないような仕組みを作る|フロントサイドで不正な入力値を弾く
データの確認・修正を簡単に行えるようにする|メニューにデータ確認・修正画面を作る


### 課題9：従業員は休暇の申請を行う時がある
対応方針|要求
----|----
打刻時と同じくらい簡単に休暇申請できるようにする|専用ページを設けて簡単にアクセスできるようにする<br>入力する情報は最低限にし、項目を選択する方式をとる<br>申請と申請状況の確認を同時にできるようにする

## 業務要件

業務区分/対象者|業務要件
----|----
QRコード送付<br>(経理担当)|データで完結し、郵送を使わずに送付できる
全従業員が<br>打刻できるようにする<br>(派遣先)|派遣元が発行した打刻用QRコードを掲示する
出退勤打刻<br>(従業員)|少ない作業で打刻する
締め日の勤怠管理<br>(経理担当)|転記等の手作業をなくす
給与計算ソフトへの<br>インポート<br>(経理担当)|CSVを既存の給与計算ソフトに連携できる形に加工する
打刻データの確認<br>(経理担当)|打刻データを一目でわかるデータにする
打刻データの修正<br>(経理担当)|誰が担当になっても簡単に修正できる<br>誰が修正したのかログを残す
休暇の申請<br>(従業員)|担当者とのやり取りを最低限に抑え、時間的・人的コストを削減する

### 画面遷移図

#### 担当者ログイン

初期登録が完了していない場合は、初期登録(会社、担当者登録)をする必要がある。

```mermaid
stateDiagram-v2
    state if1 <<choice>>
    state if2 <<choice>>
    state if3 <<choice>>

    %% 担当者　登録～ログイン
    [*] --> if1
    if1 --> 会社、担当者登録画面 : 未登録
    会社、担当者登録画面 --> if2
    if2 --> 会社、担当者登録画面 : NG(エラーメッセージ表示)
    if2 --> 各種登録完了画面 : OK
    各種登録完了画面 --> 担当者ログイン画面
    if1 --> 担当者ログイン画面 : 登録済み
    担当者ログイン画面 --> if3
    if3 --> 担当者ログイン画面 : NG(エラーメッセージ表示)
    if3 --> 担当者メニュー : OK
```

#### 申請(担当者側)

申請に対して承認/非承認を選択する操作を挟むことで、担当者・従業員の双方が申請状態を確認できるようにし、操作ログを残しやすくする。

```mermaid
stateDiagram-v2
    [*] --> 担当者メニュー
    担当者メニュー --> 申請一覧画面
    state if <<choice>>
    申請一覧画面 --> if
    if --> 休暇反映画面<br>打刻修正画面 : 申請承認
    if --> 確認ダイアログ : 申請非承認
    state if2 <<choice>>
    休暇反映画面<br>打刻修正画面 --> if2
    if2 --> 休暇反映画面<br>打刻修正画面 : NG(エラーメッセージ表示)
    if2 --> 修正・反映完了画面 : OK
    修正・反映完了画面 --> 申請一覧画面 : 5秒後遷移
    確認ダイアログ --> 申請一覧画面 : 非承認の場合、<br>申請一覧から項目を消す
```

#### 従業員打刻・ログイン

事業所QRコードからアクセスした場合のみ、打刻ボタンが使用可。<br>
直接URLを打ち込んだ場合、打刻ボタンを押せない状態にする。<br>
従業員メニューログインと共に打刻することで、従業員メニューのダッシュボードで即座に打刻状況の確認ができる。

```mermaid
stateDiagram-v2
    state if <<choice>>
    state if2 <<choice>>
    state if3 <<choice>>

    [*] --> if
    if --> ログイン画面<br>（打刻ボタン有効） : QRコード読み取り
    ログイン画面<br>（打刻ボタン有効） --> if2 : ID、パスワード入力後<br>打刻ボタン押下
    if2 --> 打刻完了画面 : OK
    打刻完了画面 --> 従業員メニュー : 5秒後遷移
    if2 --> ログイン画面<br>（打刻ボタン有効） : NG(エラーメッセージ表示)

    if --> ログイン画面<br>（打刻ボタン無効） : 直接アクセス
    ログイン画面<br>（打刻ボタン無効） --> if3
    if3 --> 従業員メニュー : OK
    if3 --> ログイン画面<br>（打刻ボタン無効） : NG(エラーメッセージ表示)
```

## 非機能要件

### 可用性

項目|説明
----|----
目標|24時間365日の稼働を目指すが、メンテナンス時は停止を許容
バックアップ|サーバー上のDBのデータを1日1回自動バックアップ

### セキュリティ

項目|説明
----|----
通信|全ての通信をHTTPS化し、外部から見られないようにする
認証|各メニューへのアクセスはID、パスワードによる認証を必須化
不正アクセス対策|認証外のアクセスはエラー画面を表示させる

### パフォーマンス

項目|説明
----|----
応答速度|打刻ボタンを押してから完了画面が出るまで3秒以内を目指す
同時接続|数名～数十名が同時に打刻してもシステムがダウンしない

### 保守性

項目|説明
----|----
ログ|打刻失敗などのエラー時に、原因特定するためのログをサーバー側に記録する
環境構築|Dockerを使用し、サーバー移行をスムーズに行えるようにする

### 拡張性

項目|説明
----|----
法改正への対応|項目の追加及び削除を容易に行えるようにする
設定の外部化|特定の計算ロジックやフォーマットをソースコードに直接書き込まず、設定ファイルやDBの定義値で管理することで、再ビルドなしで調整可能にする
異なる管理ルールへの対応|管理ルールが追加された場合でも既存のデータ構造を壊さず対応できるDB設計を維持する
