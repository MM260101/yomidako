# 「よみだこ」技術選定

本技術選定書にある選定理由は、ドキュメントや技術選定のベストプラクティスを調べながら、今の自分に必要と判断した要素を盛り込みました。現在全ての機能・技術を使いこなせている訳ではありませんが「なぜこの技術が必要なのか」を論理的に説明できることを大切にしています。

## 方針

選定の主軸は、

- 「現場の制約への適応」
- 「長期的な保守性」
- 「導入ハードルの最小化」

の3点である。

## 構成

## 開発言語・フレームワーク

### Java / Spring boot

>大規模システムや金融機関でも採用される高い信頼性と、型安全な言語特性による堅牢性を重視し採用した。「設定より規約（CoC）」を優先するSpring Bootを採用することで、開発スピードと保守性の両立を図り、将来的な担当者変更や機能拡張にも耐えうる構造とすることを目指す。<br>Javaの型チェックのお陰で記述ミスを防げた経験があり、またSpring Bootの依存関係を使用することで迷うことなくクラスを追加、DBの処理を記述できる。

### JavaScript

>フロントエンドでの入力バリデーションおよびトークンの有効性チェックを担当させる。バックエンド（API）との二重認証を挟むことで、ユーザー体験の向上と、システム全体の堅牢性を高める「多層防御」を実現する。<br>サーバーとの通信の回数を減らすことができ、その結果レスポンス速度が向上する。素早い打刻処理を行うことで、従業員になるべく負担をかけない形の打刻を実現できる。

## インフラ・実行環境

### Docker

>環境の標準化（コンテナ化）により、開発環境と本番環境の際を排除する。仮想マシンと比較して軽量かつ起動が早いため、リソースの限られた環境でも高い可用性を担保できる。<br>テスト環境をコンテナ化することで、自分のPC環境を汚すことなく「使い捨て出来る実験環境」を何度でも作り直せるのが大きな利点。本番環境であるレンタルサーバーやVPSに移行する時にも、環境の差異に起因するバグを未然に防ぎ、スムーズにサービスを開始・運用できる。

## データ管理

### PostgreSQL

>同時接続への耐性があるDBをサーバーサイドに採用することにより、出退勤時の打刻情報記録を正確に行い、データの競合も防止する。また、入力データの型チェックが厳格なため、アプリケーション層での考慮漏れによる不正なデータの混入をDBレベルで未然に防ぐねらいがある。

### SQLite

>「データを手元に置きたい」という企業の心理的ニーズに対応。サーバー側のみの集中管理ではなく、クライアント側にも軽量なファイルベースDBを持つことで、導入の容易さとデータ所有の安心感を両立させている。<br>オフライン時やサーバー停止時でも勤怠データの確認ができ、CSV出力やQRコードの発行も可能にする。また軽量であるため、クライアントアプリとあわせて持ち運びが可能。PC買い替え時の引っ越しも容易。

## 打刻・インターフェース

### QRコード(ZXing)

>派遣先に専用ハードウェアを設置できない制約をクリアするため、スマホカメラという既存のインフラを活用する。動的なトークンを含ませることで、URLの固定化やコピーによる不正打刻を技術的に防止している。<br>ワンボタンで再発行可能なため、QRコードの紛失や盗難発生時にも迅速に対応できる。
